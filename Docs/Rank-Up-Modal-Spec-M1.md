# M-1 Rank Up
## Wireframe Specification v1.0 | June 2026

**Authority:** Rank Computation Architecture (D1–D12, all locked), M-2 Honor Earned Modal Spec v1.0, M-3 Goal Achieved Modal Spec v1.0, M-4 Program Graduation Modal Spec v1.0, Ceremony Queue Amendment (June 2026), Master PRD § 13, Product DNA.

**STATUS: LOCKED — June 2026**

---

### 1. Purpose

M-1 is the ceremony moment when an athlete advances to a new rank family. It is the product's voice saying: *You have become something new.*

M-1 recognizes rank-family advancement only. Sub-rank advancement within a family does not generate M-1 and receives no ceremony. Only crossing from one rank family to the next produces M-1.

M-1 carries the highest priority in the ceremony queue. When a rank-family advancement occurs in the same session as any other recognized milestone, the athlete encounters M-1 first.

**Emotional tone:** Earned permanence and forward identity. The ceremony does not celebrate a score or reward a streak. It marks a threshold the athlete crossed by building something real. Tone combines two movements: *You earned this.* and *Welcome to what you've become.* The ceremony should feel like a recognition, not a congratulation.

---

### 2. Trigger Conditions

M-1 fires when **all** of the following are true:

1. The Rank Computation system has detected a rank-family advancement
2. A ceremony queue is active or can be opened in the current session
3. The session is live **or** a deferred ceremony from a prior offline session has been queued after synchronization

**Rank-family advancement defined:**

| Transition | M-1 fires? |
|------------|-----------|
| Foundation → Builder | Yes |
| Builder → Craftsman | Yes |
| Craftsman → Architect | Yes |
| Builder I → Builder II | No |
| Builder II → Builder III | No |

**Multiple advancements in one session:** If an import or synchronization causes multiple rank-family advancements simultaneously, one M-1 ceremony is generated per rank-family transition. Ceremonies are queued in ascending rank order and consumed sequentially.

M-1 does **not** fire when:
- The advancement is sub-rank only (within the same rank family)
- Rank computation has not completed
- The ceremony was already shown for this advancement (exactly one M-1 per rank-family transition, never re-triggered)
- A session is offline and synchronization has not yet occurred (see Edge Case E-2)

**Offline promotion:** If a rank-family advancement occurs offline, the promotion is valid immediately. The M-1 ceremony is generated after synchronization completes. The ceremony fires once and is never lost.

---

### 3. Queue Integration

M-1 is generated by the Rank Computation system and placed into the ceremony queue as a queue item. Computation and ceremony presentation are separate systems — the Rank Computation system does not present the ceremony; it generates a queue entry that the ceremony system consumes.

**Priority:** M-1 holds Priority 1 — the highest position in the ceremony queue.

```
Ceremony queue priority order:

1. M-1   Rank Up            ← highest
2. M-3   Goal Achieved
3. M-4   Program Graduation
4. M-2   Honor Earned
```

M-5 (Chapter Sealing) is a user-initiated ceremony operating in its own dedicated flow. It does not enter the auto-ceremony queue and is not subject to this priority ordering.

**Queue consumption:** The ceremony queue is consumed at defined consumption points — primarily at W-17 load (after workout completion). Other consumption points may apply when ceremonies are generated outside the workout context (import completion, goal achievement at G-2). Regardless of consumption point, queue order is always determined by the priority table above.

**Multiple M-1 ceremonies in queue:** If multiple rank-family advancements are queued (e.g., multi-rank import), each M-1 fires sequentially in ascending rank order before any M-3, M-4, or M-2 ceremonies are presented.

**Queue atomicity:** The ceremony session is atomic. No ceremony is skipped, flushed, or deferred once the queue has begun consuming. Athletes complete the active recognition sequence before entering a new workflow.

### 4. Data Contract

#### 4.1 Inputs Required

M-1 requires the following data at fire time:

| Field | Source | Required |
|-------|--------|----------|
| New rank name | Rank record | Yes |
| New sub-rank | Rank record | Yes |
| Rank badge identifier | Rank definition | Yes |
| Next rank name | Rank definition | Conditional — absent if final rank |
| Athlete display name | Athlete record | Yes |
| Forging Since date | Athlete record | Yes |
| Lifetime workout count | Athlete record | Yes |
| Days Forging (computed) | Computed from Forging Since | Yes |
| Active chapter name | Active chapter record | Conditional — absent when no active chapter |
| Ceremony trigger context | Queue metadata | Yes — for originating surface return |

#### 4.2 Outputs Written

All outputs are written before M-1 fires. The athlete's rank is updated and permanent records exist before the ceremony is presented.

**Timeline Event:** A permanent historical record is created.
- Format: `Rank Up · [Rank Name]`
- Date: advancement date (not ceremony date — relevant when deferred from offline)
- Stored at account level (not chapter-scoped)

**Share Artifact:** A data bundle is created at ceremony time for future sharing system use. No MVP sharing UI exists. The bundle captures:
- Rank Name
- Sub-Rank
- Rank badge identifier
- Athlete display name
- Forging Since date
- Advancement date

The share artifact is created regardless of whether sharing UI ever surfaces it. It is a permanent record attached to the advancement event.

M-1 does **not** create an Honor record. Rank advancement is an identity transition and a journey event — it is recognized through the Timeline Event and the ceremony itself, not through the Honors system.

#### 4.3 Final Rank Data Handling

The final rank in the system has no "Next Rank" value. The `nextRankName` field is null. M-1 content rules handle this conditionally (see Section 6.5).

### 5. Modal Structure

M-1 is a centered modal overlay. The underlying surface (typically W-17) is visible but dimmed behind M-1 until dismissed.

**Standard rank advancement:**

```
┌──────────────────────────────────────┐
│                                      │
│           [ RANK BADGE ]             │  ← Badge, prominent, centered
│                                      │
│              Architect               │  ← Rank Name, large, centered
│                 IV                   │  ← Sub-Rank, below rank name
│                                      │
│    Earned through every session.     │
│    Welcome to what you've become.    │  ← Ceremony copy, centered
│                                      │
│  ──────────────────────────────────  │
│                                      │
│  Forging Since     June 2024         │
│  Total Workouts    247               │
│  Days Forging      389               │
│  Current Chapter   Road to 405       │  ← Omitted when no active chapter
│                                      │
│  Next Rank: Legacy                   │  ← Directional only, no numerics
│                                      │
│           [ Continue ]               │  ← Primary CTA
│                                      │
│   View Your Rank    View Your Legacy │  ← Conditional text links
│                                      │
└──────────────────────────────────────┘
```

**Final rank variant:**

The final rank in the system uses elevated presentation in place of the standard layout:

```
┌──────────────────────────────────────┐
│                                      │
│        [ UNIQUE RANK ARTWORK ]       │  ← Unique artwork, not standard badge
│                                      │
│           [Final Rank Name]          │  ← Rank Name, large, centered
│                                      │
│    "Your legacy has been forged."    │  ← Final rank ceremony copy, locked
│                                      │
│  ──────────────────────────────────  │
│                                      │
│  Forging Since     June 2024         │
│  Total Workouts    247               │
│  Days Forging      389               │
│  Current Chapter   Road to 405       │  ← Conditional
│                                      │
│  [Next Rank row absent]              │  ← No next rank — row omitted
│                                      │
│           [ Continue ]               │
│                                      │
│   View Your Rank    View Your Legacy │  ← Conditional text links
│                                      │
└──────────────────────────────────────┘
```

The final rank shares the same architecture, queue behavior, persistence model, and CTA rules as standard rank advancement. Only the presentation layer differs. Specific artwork and badge treatment are design deliverables outside the scope of this wireframe specification.

### 6. Content Rules

#### 6.1 Rank Name and Sub-Rank

- Rank Name: displayed at large primary weight, centered. Sourced from the rank record of the newly attained rank.
- Sub-Rank: displayed below Rank Name at secondary weight. Sourced from the rank record.
- Both fields are always present for standard rank advancement.
- The Rank Name displayed is the new rank — the rank the athlete just entered, not the rank just left.

#### 6.2 Rank Badge

- Badge is sourced from the rank definition for the newly attained rank family.
- Displayed prominently, centered, above the rank name.
- No icon, star, trophy, or gamification imagery. The badge is the rank's own visual identity.
- Final rank: unique artwork replaces the standard badge (see Section 5).
- Badge is decorative identity — not tappable.

#### 6.3 Ceremony Copy

Standard ceremony copy (all non-final ranks):

> *"Earned through every session. Welcome to what you've become."*

This copy:
- Is rank-agnostic — no variations per rank family
- Does not say "Congratulations"
- Does not reference XP, points, or scores
- Combines earned acknowledgment with forward identity welcome
- Is calm, not effusive — recognition, not spectacle

Final rank ceremony copy (locked):

> *"Your legacy has been forged."*

#### 6.4 Journey Context

The journey context section displays four rows. Each row is static text — none are tappable.

| Row | Label | Value | Condition |
|-----|-------|-------|-----------|
| 1 | Forging Since | Month YYYY (e.g., "June 2024") | Always shown |
| 2 | Total Workouts | Integer count | Always shown |
| 3 | Days Forging | Integer count, computed from Forging Since date | Always shown |
| 4 | Current Chapter | Active chapter name | Shown only when active chapter exists |

When no active chapter exists, the Current Chapter row is omitted entirely. No placeholder text. No "No Chapter" label. The section renders with three rows.

Numeric values (Total Workouts, Days Forging) are displayed as plain integers — no formatting units ("workouts", "days") are shown inline in the value column. Labels carry the meaning.

#### 6.5 Directional Progression

One line below the journey context section displays the name of the next rank:

> Next Rank: Legacy

Rules:
- Shows the rank-family name only — no sub-rank suffix, no XP values, no percentages, no progress bars
- Sourced from the rank definition's `nextRankFamily` field
- **Omitted entirely when M-1 fires for the final rank.** No "You've reached the top" replacement label. The row is absent.

#### 6.6 Final Rank Exception

When M-1 fires for the final rank in the system:
- Rank Badge → unique artwork (design deliverable)
- Ceremony copy → "Your legacy has been forged." (locked)
- Directional Progression row → omitted
- Timeline styling → unique (engineering deliverable per rank definition)
- Share artifact → unique styling applied to bundle (for future sharing system)
- All other content rules apply unchanged

### 7. CTA Rules

#### 7.1 Primary CTA

`"Continue"` — centered, primary style. Always present. Always the only button-class element.

#### 7.2 Secondary Navigation Links

Two text links appear below the primary CTA. They are **not buttons** — small text link style only, no visual competition with Continue.

| Link | Destination |
|------|------------|
| `"View Your Rank"` | P-3 Rank Detail |
| `"View Your Legacy"` | L-1 Legacy Hub |

#### 7.3 Queue-Conditional Display

Secondary navigation links are displayed only when M-1 is the final remaining ceremony in the active queue.

| Queue state | CTA display |
|-------------|-------------|
| Additional ceremonies remain after M-1 | `"Continue"` only |
| M-1 is the final ceremony | `"Continue"` + `"View Your Rank"` + `"View Your Legacy"` |

When a secondary link is tapped, M-1 dismisses and the athlete navigates to the link destination. The ceremony session is complete — secondary link navigation is only available when no ceremonies remain.

---

### 8. Navigation Rules

| Action | Destination |
|--------|------------|
| `"Continue"` — ceremonies remain in queue | Next queued ceremony (M-3, M-4, or M-2 per priority) |
| `"Continue"` — M-1 is final ceremony | Originating surface (surface active when queue was consumed) |
| `"View Your Rank"` tap | P-3 Rank Detail (M-1 dismissed) |
| `"View Your Legacy"` tap | L-1 Legacy Hub (M-1 dismissed) |
| Tap outside modal | No action — M-1 remains open |
| Drag gesture | No action — M-1 is a centered modal, not a bottom sheet |
| Android hardware back | No action — M-1 remains open |

**Originating surface:** The surface that was active when the ceremony queue began consuming. For workout-triggered ceremonies, this is W-17. For import-triggered ceremonies, this is the import completion screen. After all queued ceremonies complete, the originating surface is fully revealed.

**Multiple M-1 ceremonies:** When multiple M-1 ceremonies are queued (multi-rank advancement), each M-1 fires and dismisses via `"Continue"` before the next M-1 appears. Secondary links are suppressed for all M-1 ceremonies except the final item in the overall queue.

### 9. Accessibility

- Modal announced to screen reader as `"Rank up"`
- Rank Name announced at heading level
- Sub-Rank announced as secondary heading or as part of heading group
- Rank Badge announced as `"[Rank Name] badge"` (decorative but named)
- Ceremony copy announced as static text
- Journey context rows announced as static text (label-value pairs)
- `"Next Rank: [Name]"` announced as static text when present
- `"Continue"` CTA is the default focus target when modal opens
- Focus is trapped within the modal while open (standard modal behavior)
- After dismissal, focus returns to the originating surface
- Secondary navigation links, when visible, are announced as links: `"View Your Rank, double-tap to navigate"` / `"View Your Legacy, double-tap to navigate"`
- Minimum touch target: 44×44pt for `"Continue"` and all secondary link rows

### 10. Edge Cases

**E-1: Multi-rank import progression**
If an athlete imports historical training data and the Rank Computation system detects multiple rank-family advancements, one M-1 ceremony is generated per transition. Ceremonies are queued in ascending rank order. Each fires and is dismissed via `"Continue"` before the next appears. Secondary navigation links are suppressed until the final ceremony in the overall queue. No consolidation. No ceremony is skipped.

**E-2: Offline promotion synchronization**
If a rank-family advancement occurs while the athlete is offline, the promotion is applied immediately to the local record. M-1 is generated after synchronization completes. The ceremony fires once at the next natural queue consumption point after sync. The ceremony is never lost. The Timeline Event uses the advancement date, not the ceremony date.

**E-3: Multiple rank-family promotions in a single day**
If an athlete reaches two rank families in a single day (e.g., two separate workout sessions), two M-1 ceremonies are generated. Each is queued and consumed at its natural queue point (end of the session that triggered each advancement). They do not merge. Each ceremony fires with accurate rank data for that specific transition.

**E-4: No active chapter**
When M-1 fires and no active chapter exists, the "Current Chapter" row in the journey context section is omitted. No placeholder text. No fallback label. The modal is valid and complete with three journey context rows.

**E-5: M-1 is the final ceremony in the queue**
`"Continue"` button and secondary navigation links (`"View Your Rank"` + `"View Your Legacy"`) are both displayed. Tapping `"Continue"` returns to the originating surface. Tapping either secondary link navigates directly to that destination and completes the ceremony session.

**E-6: Additional ceremonies remain in queue after M-1**
Secondary navigation links are not displayed. Only `"Continue"` is shown. Tapping `"Continue"` advances to the next ceremony in priority order (M-3 → M-4 → M-2).

**E-7: Final rank advancement**
M-1 fires with elevated presentation: unique artwork replaces the standard badge, ceremony copy reads "Your legacy has been forged.", and the `"Next Rank"` row is absent. All other behavior — queue, persistence, CTA rules, navigation, accessibility — is identical to standard rank advancement.

---

### 11. Non-Behaviors

M-1 does not and will never:

- Fire for sub-rank advancement within the same rank family
- Create an Honor record
- Trigger M-2 (Honor Earned) as a downstream effect
- Display XP values
- Display percentages
- Display progress bars
- Display the previous rank
- Display "Congratulations" or effusive language
- Display gamification language (points, scores, streaks)
- Display rank contribution from specific workouts or goals
- Display sharing UI in MVP (share artifact is created but no sharing surface exists)
- Fire mid-session (W-9–W-16)
- Fire as a push notification
- Fire at app open, home screen load, or tab switch independently of a ceremony queue
- Dismiss on tap outside the modal
- Dismiss on drag gesture
- Dismiss on Android hardware back
- Re-trigger after dismissal for the same rank-family advancement
- Consolidate multiple rank-family advancements into a single ceremony
- Suppress or skip any queued ceremony
- Persist queue state beyond the active session
